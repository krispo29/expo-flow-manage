# คู่มือการใช้งาน Sentry ขั้นพื้นฐาน (Sentry Guide)

## 1. Sentry คืออะไร?

Sentry เป็นเครื่องมือสำหรับเฝ้าระวัง (Monitoring) และติดตามข้อผิดพลาด (Error Tracking) แบบเรียลไทม์ มันช่วยให้เรารู้ทันทีเมื่อมีข้อผิดพลาดเกิดขึ้นในแอปพลิเคชัน (ทั้งฝั่ง Frontend, Backend API และ Database Connection) โดยไม่ต้องรอให้ผู้ใช้งานแจ้งเข้ามา

## 2. ฟีเจอร์หลักที่มีประโยชน์มากๆ

- **Error Tracking**: ตรวจจับและบันทึกข้อผิดพลาด พร้อมแจ้งเตือนผ่านช่องทางต่างๆ (เช่น Email, Slack, Microsoft Teams, Discord)
- **Stack Traces & Source Maps**: ชี้เป้าได้ว่า Error เกิดขึ้นที่ไฟล์อะไร บรรทัดไหน โค้ดส่วนไหน (แม้โค้ดจะถูก Build จนบีบอัดไปแล้วก็ตาม)
- **Breadcrumbs**: บันทึกเหตุการณ์ (สเต็ปการทำงาน) ก่อนเกิด Error ว่าผู้ใช้คลิกอะไร พิมพ์อะไร หรือแอปทำงานฟังก์ชันไหนไปบ้าง ถึงนำมาสู่ความผิดพลาด
- **Session Replay**: สามารถดูวิดีโอย้อนหลัง จำลองจากหน้าจอผู้ใช้จริงๆ ในช่วงเวลาที่เกิด Error (บันทึกแค่พฤติกรรม ไม่ได้อัดวิดีโอจากอุปกรณ์จริงๆ) ทำให้เห็นภาพชัดเจนว่าเกิดอะไรขึ้น
- **Performance Monitoring (Tracing)**: ตรวจจับระยะเวลาการทำงานของ API ว่าคอขวดอยู่จุดไหน หรือคิวรี่จากฐานข้อมูลบ่อยหรือช้าเกินไปหรือไม่

## 3. การใช้งาน Sentry ในโปรเจกต์ Next.js ของเรา

โดยปกติแล้ว Next.js SDK ของ Sentry ระบบจะจับ **Uncaught Exceptions (Error ที่ไม่ได้ดักจับ)** ให้โดยอัตโนมัติอยู่แล้ว แต่ถ้าต้องการเขียนส่งเองแบบ Manual สามารถใช้ตัวอย่างดังนี้:

### 3.1 การจับ Error เอง (กรณีอยู่ใน try...catch)

เวลาคุณเรียกใช้ API หรือฟังก์ชันที่สำคัญ ถ้ามีการดักจับข้อผิดพลาดอยู่ ควรให้มันส่งรายงานเข้า Sentry ด้วย

```typescript
import * as Sentry from '@sentry/nextjs'

try {
  // โค้ดที่อาจจะเกิดปัญหา
  const data = await fetchExhibitorData()
} catch (error) {
  // ส่งข้อผิดพลาดไปที่ Sentry พร้อมข้อมูล Stack trace
  Sentry.captureException(error)

  // จากนั้นก็จัดการโชว์ UI เตือน หรือ Response Error ให้ User กลับไป
}
```

### 3.2 การส่งข้อความแบบแมนนวล (Custom Messages)

เพื่อบันทึกข้อความเฉพาะในบางกรณีที่ไม่ใช่ Error ร้ายแรง แต่เราอยากรู้:

```typescript
Sentry.captureMessage('มีผู้ใช้พยายามข้ามขั้นตอน Profile Setup', 'info')
// ระดับของความรุนแรง: "info", "warning", "error", "fatal", "debug"
```

### 3.3 การบันทึกข้อมูลผู้ใช้ (User Context)

หากทำระบบ Login เสร็จแล้ว (เช่น รู้แล้วว่า Admin หรือ Organizer ล็อกอิน) เราสามารถแนบประวัติผู้ใช้ไป Sentry ได้เลย ทำให้เวลาเกิดปัญหาเรารู้ว่า **"ใคร"** เป็นคนเจอ:

```typescript
Sentry.setUser({
  id: 'user_123', // UUID จัดเก็บในระบบ
  email: 'admin@example.com', // อีเมลผู้ใช้งาน
  username: 'admin_tester',
  segment: 'organizer', // แนบข้อมูลอะไรมาก็ได้ที่คุณอยากเห็น
})

// ข้อควรระวัง: ควรทำการ clear user ทิ้งเมื่อผู้ใช้ออกจากระบบ (Logout)
// Sentry.setUser(null);
```

### 3.4 การตั้งค่าโหมด (Environments) ให้แสดงผลแยกกัน

ตอนนี้ Sentry น่าจะดึงค่า `NODE_ENV` ให้อัตโนมัติ (เป็น "production", "development") เพื่อให้ใน Dashboard เรากรองดูได้ว่าบั๊กนี้มาจากใคร

## 4. ปัญหาที่ต้องระวัง (Troubleshooting)

### ปัญหา `npm run build` ไม่ผ่านหลังติดตั้ง (ปัญหาจากโฟลเดอร์แคช `.next`)

หลังจากที่คุณสั่งลบตัวอย่าง `sentry-example-page` ออก แล้วพอรัน `npm run build` ทันที มันอาจจะขึ้น Error สีแดงแจ้งว่าพังตรง **Type Declaration** (`__IsExpected<Specific extends AppPageConfig>...` แบบที่คุณพบเจอ)
**สาเหตุ:** เนื่องจาก Next.js ทำการบิ้วแคชไฟล์ (Typescript) ตอนที่เราเปิด `npm run dev` ค้างไว้ พอเราลบไฟล์ออกไป แล้วมารันบิ้ว แคชตัวเก่ายังเรียกหาไฟล์ที่เราพึ่งลบทิ้งไป
**วิธีแก้:** ให้ลบโฟลเดอร์แคช `.next` ออกก่อน 1 สเต็ป

```bash
Remove-Item -Recurse -Force .next
npm run build
```

## 5. เรื่องความปลอดภัยและข้อมูลส่วนตัว (Data Privacy & PII)

- Sentry จะตั้งค่าระบบให้กรองข้อมูลสำคัญ (Scrub Data) อย่างเช่น **รหัสผ่าน**, **คีย์เข้ารหัสลับ**, ทิ้งก่อนที่จะส่งมาบน Dashboard
- แนะนำว่าในฟังก์ชันดักจับ Error ของคุณ **อย่าส่งตัวแปร (Variables)** ที่เป็นความลับไปดื้อๆ ให้ Sentry เช่น บัตรประจำตัวประชาชน ตัวเลขบัตรเครดิต เพราะมันจะไปกระจายโชว์ในวงผู้พัฒนาทุกคนที่มีสิทธิ์เห็น Dashboard
